# CLAUDE.md

## 作業開始時のチェックリスト

1. `docs/product.md` を読みプロダクトを理解する
2. `docs/requirements.md` で今回のタスクに関連する要件を確認する
3. `docs/architecture.md` で実装方針を確認する
4. `docs/tasks.md` で実装順を確認し、タスクの状態は GitHub Issues で確認する

---

## ドキュメント構成ルール

実装・コード生成・DB設計を行う前に、以下のドキュメントが `docs/` 配下に揃っていることを確認すること。
不足しているドキュメントがある場合は、作業開始前にユーザーに確認または作成を促す。

### 必須ドキュメント

| ファイル | 役割 |
|----------|------|
| `docs/product.md` | プロダクトの目的・対象ユーザー・ゴール |
| `docs/requirements.md` | 機能要件・非機能要件の一覧 |
| `docs/architecture.md` | 技術スタック・構成・実装方針 |
| `docs/api.md` | APIエンドポイント定義（メソッド・パス・リクエスト/レスポンス形式） |
| `docs/schema.md` | DBテーブル定義・リレーション・インデックス |
| `docs/tasks.md` | フェーズ構成の設計図（各フェーズのマイルストーンへのリンク集。タスク詳細・状態は GitHub Issues で管理） |

### 追加ドキュメント（必要に応じて作成）

以下のケースでは追加ドキュメントを `docs/` に作成する：

- 認証・認可フローが複雑な場合 → `docs/auth.md`
- 外部サービス連携（Slack, Email等）がある場合 → `docs/integrations.md`
- UIコンポーネント設計が必要な場合 → `docs/ui.md`
- デプロイ・インフラ構成が必要な場合 → `docs/infra.md`
- テスト方針が必要な場合 → `docs/testing.md`

## ドキュメント更新ルール

- 実装中に仕様変更が発生した場合、コード修正と同時に該当ドキュメントを更新する。
- `tasks.md` は各フェーズのマイルストーンへのリンク集として維持する。個別タスクの詳細・T番号・完了状態は記載せず、GitHub Issues で管理する。
- ドキュメントとコードの乖離が生じた場合は、原則ドキュメントを正として扱いコードを修正する。ただし意図的な仕様変更の場合はドキュメントを先に更新してからコードを変更する。

## テストルール

- **APIルートの実装はユニットテストの作成をもって完了とする**
- **`src/lib/` 配下のユーティリティ関数もユニットテストの作成をもって完了とする**
- UIコンポーネントのユニットテストは必須としない（E2E やビジュアルリグレッションで担保する場合は除く）
- テストファイルは実装ファイルと同じディレクトリに `[name].test.ts` で配置する
- Prisma・外部サービス（bcrypt 等）は `vi.mock` でモック化する
- 最低限カバーすべきケース:
  - 正常系（期待するステータスコードとレスポンス）
  - バリデーションエラー（400）
  - 認可エラー（403）※認可チェックがある場合
  - リソース重複・未存在（409 / 404）※該当する場合
- ユーティリティ関数で最低限カバーすべきケース:
  - 正常系（期待する戻り値）
  - 境界値・エッジケース（空文字、フォーマット違反、範囲外の値など）
- テストファイルの先頭に `// @vitest-environment node` を付ける（APIルートは Node 環境で実行）

## UI実装の動作確認ルール

- **UIを含む実装はアプリを起動した手動動作確認をもって完了とする**
- 実装完了後、コミット・プッシュの前に、動作確認の内容・観点・手順を一覧化してユーザーに提示する
- ユーザーが動作確認を実施し、問題がなければコミット・プッシュに進む

### 動作確認チェックリストの形式

以下の観点を網羅した表形式で提示する：

| # | 手順 | 確認観点 | 期待値 |
|---|---|---|---|

### 最低限カバーすべき観点

- **認証・認可**: 未ログイン時のリダイレクト、ログイン成功・失敗
- **画面表示**: データが正しく表示されるか、空状態の表示
- **インタラクション**: ボタン・フォームの操作、状態変化（選択・入力）
- **保存・更新**: 保存後のフィードバック表示、リロード後のデータ永続化
- **ユーザー分離**: 複数ユーザーでデータが混在しないか（該当する場合）

### 起動コマンド（devcontainerで実行）

```bash
npm run dev
```

## コーディングルール

- 言語: TypeScript（strict モード）
- パッケージマネージャ: npm
- フォーマッタ・リンター: Biome（`biome.json` の設定に従う）
- コメント: 自明でないロジックにのみ付ける（過剰なコメント不要）
- エラーハンドリング: 外部入力・APIレスポンスの境界でのみ行う

## Prisma スキーマのカラム名ルール

- Prisma フィールド名は **camelCase**（TypeScript との一貫性）
- DB カラム名は **snake_case**（PostgreSQL の慣習）
- 必ず `@map("snake_case_name")` で明示的にマッピングする

```prisma
// 例
clerkId      String?  @unique @map("clerk_id")
isActive     Boolean  @default(true) @map("is_active")
createdAt    DateTime @default(now()) @map("created_at")
invitedById  String   @map("invited_by_id")
```

## GitHub Issues 管理ルール

新規タスク・不具合はすべて GitHub Issues で管理する。

### Issue 作成ルール

| 項目 | ルール |
|------|--------|
| タイトル | `[T番号] タスク名` の形式（例: `[T60] XXX機能追加`） |
| T番号 | 直前の最大番号 + 1 で採番する |
| Milestone | 対象フェーズの Milestone をアサインする |
| Label | `enhancement`（新機能）/ `fix`（不具合）/ `testing`（テスト）から選ぶ |
| 本文 | 概要・要件・依存 Issue を記載する |

### Issue クローズルール

Issue をクローズする際は、**必ず関連するコミット ID をコメントに含める**。

```
gh issue close <番号> --comment "完了。実装コミット: <short-hash> (<コミットメッセージ>)

<補足説明（任意）>"
```

## バージョニングルール

`package.json` の `version` はセマンティックバージョニング（SemVer）に従う。
**`develop` ブランチへのマージ直前**に、`develop` ブランチの現在バージョンを確認したうえでバージョンを決定し、更新する。

| 変更の種類 | バージョン操作 | 例 |
|-----------|-------------|-----|
| 新機能追加（フェーズ完了など） | マイナー +1、パッチ → 0 | `0.1.0` → `0.2.0` |
| バグ修正・小改善のみ | パッチ +1 | `0.2.0` → `0.2.1` |
| 破壊的変更（APIや DB スキーマの非互換変更） | **ユーザーの明示的な指示のみ** | `0.x.x` → `1.0.0` |

- `1.0.0` は本番リリース（MVP 完成）など、ユーザーが指定したタイミングで上げる
- バージョン更新のコミットメッセージは `chore: bump version to x.x.x` とする

## Git 操作ルール

- 実装・ドキュメント更新が完了したら変更内容をユーザーに提示し、**明示的な承認（「OK」など）を受けてから** `git commit` / `git push` を実行する

### ブランチ・マージルール

- `develop` および `master` へのマージは **必ず PR を作成**し、ユーザーの承認を得てからマージする
- 直接マージ（`git merge`）は行わない

### Issue 単位の作業フロー

1 つの Issue に対応する実装が完了したら、以下の順で処理する：

1. 変更内容をユーザーに提示して承認を待つ
2. 承認後、`git commit` → `git push` を実行する
3. 対応する GitHub Issue を関連コミット ID を添えてクローズする
4. 次の Issue に進んでよいか確認をとる

### 作業スコープルール

- **指示されたタスク（Issue）のみを実行する**。ユーザーが明示的に指定していないタスクは、自己判断で着手しない。
- 複数 Issue をまとめて依頼された場合も、1 Issue ごとに完了・承認を経てから次に進む。
- 「マイルストーン X を進める」など範囲指定の指示を受けた場合は、着手前に対象 Issue の一覧と実施順をユーザーに提示して確認をとる。

## PR レビュー指摘の対応ルール

- Copilot・レビュアーからの指摘は、マージ前に同一ブランチ内で対応する
- 対応不要と判断した場合はその理由をコメントで返す
- 指摘対応のコミットメッセージは `fix: <内容>` とする
- 指摘に対応したら、該当コメントに対応済みの旨をコミット ID を添えて返信し、スレッドを Resolve する

```
対応済み。修正コミット: <short-hash> (<コミットメッセージ>)

<補足説明（任意）>
```
